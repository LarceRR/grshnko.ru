Universal ESP32 WS2812B Animation Engine - Full Implementation Plan
Project Overview
Create a universal, high-performance firmware for ESP32-DevKitC-32 (ESP32-WROOM-32, 4 MB flash) that drives a WS2812B LED strip (300 LEDs, 60 LEDs/m, linear layout only). The core is a powerful animation engine capable of interpreting complex animations generated by an LLM in a custom JSON format. The system must be fully controllable via an existing Node.js + Express + WebSocket backend. All existing non-animation logic (Wi-Fi connection, WebSocket reconnection, OTA updates, progress bar on the strip, connection indicators, and "Engine Started" indicator at the end of the strip) must be preserved untouched.
The new animation engine replaces only the old animation system. The rest of the firmware stays exactly as it is.
Hardware and Existing Infrastructure

ESP32-WROOM-32 (520 KB SRAM, 4 MB flash)
WS2812B linear strip, 300 LEDs
Existing Node.js backend with WebSocket server
Existing Prisma + MongoDB database (all animation-related collections will be cleared to zero by the user)
OTA update mechanism must remain fully functional
Progress bar on the strip during OTA must remain
Wi-Fi connection indicator and "Engine Started" indicator at the end of the strip must remain

Animation JSON Format (Exact Specification)
The animation is a single JSON object with the following root fields (all required unless noted):

version: 1 (integer)
title: string (auto-generated by server if missing)
description: string (empty string if missing)
author: string (MongoDB ObjectId as string, added by server)
generatedBy: string (LLM model identifier, added by server)
generationPrompt: string (exact prompt sent to LLM, added by server)
labels: array of strings (4–10 tags, lowercase, generated by LLM)
usedColors: array of strings in #rrggbb format only (3–15 unique colors, generated by LLM)
fps: integer (default 60)
loop: boolean
master_brightness: float 0.0–1.0 (default 0.85)
master_opacity: float 0.0–1.0 (default 1.0)
params: object containing only numbers, hex color strings, and one optional "palette" array of hex strings
paramDescriptions: object (keys match params, each value is an object with at least "description", optional "effect", "range", "recommended")
layers: array of maximum 6 layer objects

Each layer object contains:

name: string (optional)
opacity: float 0.0–1.0
blend: string (one of: "normal", "add", "screen", "multiply", "overlay", "lighten")
expr: string containing a functional expression that returns either a color or an object { color: any-color-format, brightness: 0.0–1.0 }

Color formats supported everywhere: "#rrggbb", [r,g,b] 0–255 or 0–1, {h: 0–360 or 0–1, s: 0–1, v: 0–1}, rgb(r,g,b), hsv(h,s,v)
Expression Language (expr)
Functional style only. Supported built-in functions (exact list, no more, no less):
Mathematical: sin, cos, tan, asin, acos, atan, atan2, abs, floor, ceil, round, fract, pow, sqrt, min, max, clamp, lerp, map
Logical: if(cond, then, else), select, ternary ? :
Geometry: distance, dist_from_center
Time/Rhythm: time (global seconds), beat(bpm, duty), pulse, fract
Color: hsv, rgb, hsv_bright, rgb_bright, blend(color1, color2, amount, mode)
Palette: palette_index(t, palette)
Noise: perlin_noise(x, y, z, scale), random, random(min, max)
Ease: linear, easeInQuad, easeOutQuad, easeInOutQuad, easeOutExpo, easeOutBounce
All expressions are evaluated per LED per frame. pos is always 0.0–1.0 (normalized). The engine must safely handle parse and runtime errors without crashing the device.
WebSocket Protocol (Exact)
All messages are JSON objects with a required "type" field.
Server → Device commands:

animation_load → full animation JSON
params_update → { params: {…}, transition_time?: number (default 0.5) }
master_update → { master_brightness?: number, master_opacity?: number }
pause
resume
stop (return to fallback)
ping

Device → Server responses:

connected (on connect)
status (periodic + on state change)
ack (for every command)
error (detailed stage, layer, message)

Pause freezes time but keeps the current frame displayed. Resume continues from the exact same time. During pause the strip can drop to very low refresh rate for power saving.
Parameters are interpolated smoothly over the transition_time using easeInOutQuad.
Fallback / Error Behavior

No valid animation ever loaded → entire strip is warm orange (#FF6200) with slow 4-second easeInOutQuad pulse at 30 fps
Critical engine failure → same pulse but bright red (#FF1A1A)
Partial animation load → render only successfully parsed layers
Runtime errors in expr → black pixel for that LED only, layer continues

Database & Prisma Requirements
The agent must completely rewrite the Prisma schema for the new animation system. Existing animation collections will be cleared to zero by the user. The new schema must include:

Animation model with all root fields (title, description, labels, usedColors, etc.)
Full JSON storage of the original animation for audit
User relation via author ObjectId
Timestamps
Indexes for fast search by labels, usedColors, author, generatedBy

The backend must continue to use the existing Prisma client after schema update.
Code Preservation Requirements (Critical)
The development agent MUST NOT touch or remove any of the following existing code:

All OTA update logic and progress bar rendering on the strip during OTA
All Wi-Fi connection indication logic and LED patterns
All WebSocket connection, reconnection, and heartbeat logic
The "Engine Started" indicator at the end of the strip
All device initialization, configuration loading, and non-animation related tasks

Only the old animation rendering system may be replaced by the new engine.
Implementation Order (Strict)

Core expression parser and evaluator (safe, fast, per-LED)
Layer rendering pipeline with blending and opacity
Full animation JSON loader and validator
Parameter interpolation system with transition_time
WebSocket command handler for all new message types
Fallback animation (orange/red pulse)
Error handling and detailed reporting over WS
Integration with existing FastLED/NeoPixelBus setup
Prisma schema update and backend adjustments
Full testing with real LLM-generated animations

The final firmware must run at 60 fps reliably with up to 6 layers and 12 KB JSON on the target ESP32 hardware.
